## Decision: EAS Build, not bare workflow

Two options for shipping an Expo app:

| Option | Pros | Cons |
|---|---|---|
| **Expo Go + EAS Update** | Free, instant updates, no builds | Limited to Expo SDK modules; can't use arbitrary native code |
| **EAS Build → TestFlight / Play** | Full native capability, real app store presence | Build time, code signing, costs after free tier |

**Choice: EAS Build.** Because:
- We need push notifications via Expo Notifications (works in dev but full
  config requires a real binary)
- We need a real bundle ID for App Store + Play Store listings
- We will likely add Sentry, which requires a build to do source maps
  correctly
- Free tier (30 builds/month) is enough for solo dev pace

## EAS profiles — planned

`eas.json` will define three profiles:

| Profile | Purpose | Distribution |
|---|---|---|
| `development` | Dev client with debugger | Internal (Expo Go-like) |
| `preview` | Internal test build, prod env | Internal distribution + TestFlight |
| `production` | App Store + Play Store releases | Public stores |

Build commands (after Phase 4 setup):

```bash
eas build --profile development --platform ios
eas build --profile preview --platform all
eas build --profile production --platform all
```

## Code signing

iOS:
- Apple Developer account needed (€99/year, registered under Tailor or Memora SRL)
- EAS handles certificate + provisioning profile generation
- Distribution certificate stored in EAS Secrets

Android:
- Google Play Developer account needed ($25 one-time)
- Upload key generated by EAS, stored in EAS Secrets
- Once configured, Play handles app signing on its side

## Environment variables in builds

EAS Build needs production values for `EXPO_PUBLIC_SUPABASE_URL` and
`EXPO_PUBLIC_SUPABASE_ANON_KEY`. Two options:

1. Per-profile `env` in `eas.json` (committed, but contains only public keys)
2. EAS Secrets (encrypted, set via `eas secret:create`)

Choice: option 1 for `EXPO_PUBLIC_*` (they're already public in the bundle),
option 2 for any server-only values that get added later (none in Phase 1-3).

## OTA updates (`expo-updates`)

Decision: **enable only for the `production` channel, after first launch**.

Why:
- Lets us push bug fixes without an app store review for JS-only changes
- We avoid using it during Phase 4 testing to keep version control honest

Configuration: `expo-updates` package + `app.json` `updates` block. Documented
in detail when we set it up.
